# English Feedback Flow

This document describes the current feedback-generation flow in `WriteLoop` after the move to the `fixPoints[]`-first UI contract.

## 1. Current North Star

The live feedback screen is built around a small set of UI-ready outputs:

- `strengths`
- `ui.fixPoints`
- `ui.nextStepPractice`
- `ui.rewriteSuggestions`
- `modelAnswer`
- `usedExpressions`

The main correction flow is no longer designed around a `primaryFix + secondaryLearningPoints` split. Instead, the model is asked to produce one ordered list:

- `fixPoints[0]`: the first correction the learner should focus on
- `fixPoints[1...]`: the remaining distinct, lower-priority fixes

Each `fixPoints` item should teach exactly one correction point.

## 2. Request Flow

`apps/frontend/app/answer-loop.tsx`

1. The learner submits an answer.
2. The frontend calls `/api/feedback`.
3. `FeedbackService.review()` loads prompt/session context and delegates to `LlmFeedbackClient`.
4. `GeminiFeedbackClient.reviewHybrid()` runs:
   - diagnosis
   - generation
   - sanitization / validation
   - fallback merge when needed
   - final response assembly
5. `FeedbackUiComposer.compose()` creates the final UI blocks consumed by the frontend.

## 3. Diagnosis Stage

Diagnosis is still separate from generation. Gemini first returns:

- `answerBand`
- `taskCompletion`
- `onTopic`
- `finishable`
- `grammarSeverity`
- `minimalCorrection`
- `primaryIssueCode`
- `secondaryIssueCode`
- `rewriteTarget`
- `expansionBudget`
- `regressionSensitiveFacts`
- `grammarIssues`

Diagnosis decides the teaching direction. Generation then turns that direction into UI-ready sections.

## 4. Internal Generation Contract

The structured Gemini generation step is now centered on an internal generation contract:

- `strengths`
- `fixPoints`
- `grammarFeedback`
- `corrections`
- `secondaryLearningPoints`
- `usedExpressions`
- `refinementExpressions`
- `nextStepPractice`
- `rewriteSuggestions`
- `modelAnswer`
- `modelAnswerKo`

The public learner-facing contract is smaller:

- `strengths`
- `ui.fixPoints`
- `ui.nextStepPractice`
- `ui.rewriteSuggestions`
- `modelAnswer`
- `usedExpressions`

### 4.1 UI-ready fields

These are intended to be close to directly renderable in the public feedback UI:

- `fixPoints`
- `nextStepPractice`
- `rewriteSuggestions`
- `modelAnswer`

### 4.2 Raw candidate pools

These are still generated internally, but they mainly act as supporting material:

- `grammarFeedback`
- `corrections`
- `refinementExpressions`

`grammarFeedback` is no longer part of the public learner-facing response contract. It exists only as internal raw grammar material that can support validation, fallback, and fix-point assembly.

### 4.3 Compatibility-only fields

These fields still exist in parts of the backend, but they are no longer part of the active Gemini generation contract for the main feedback screen:

- `ui.focusCard`
- `ui.primaryFix`

Generation should not create them anymore. The system derives any needed compatibility information from `fixPoints`.

## 5. Legacy Fields That Still Exist

Some old fields still exist because other parts of the product use them.

### 5.1 `summary`

`summary` is no longer part of the active Gemini feedback-generation contract for the main feedback screen.

It still exists as a legacy field for:

- persistence
- history views
- compatibility with older flows

### 5.2 `inlineFeedback`

`inlineFeedback` is also not generated by Gemini as a first-class prompt output.

It is still kept for:

- history / review surfaces
- derived diff-style feedback
- downstream compatibility

In practice, it is built from corrected-answer style comparisons rather than treated as a primary feedback-screen artifact.

### 5.3 `ui.focusCard`

`ui.focusCard` is deprecated for the main feedback screen:

- it is no longer rendered in the main feedback view
- it is hidden from frontend serialization in `FeedbackUiDto`
- it is no longer part of the active Gemini generation prompt/schema

### 5.4 `ui.primaryFix`

`ui.primaryFix` is also deprecated for the main feedback screen:

- the frontend uses `ui.fixPoints` instead
- it is hidden from frontend serialization in `FeedbackUiDto`
- it is no longer part of the active Gemini generation prompt/schema

Internally, some compatibility logic may still derive a `primaryFix` view from `fixPoints[0]`.

## 6. Parsing and Sanitization

`GeminiFeedbackClient.parseGeneratedSections(...)`

Current parser behavior:

- parse `fixPoints`
- derive compatibility `primaryFix` from `fixPoints[0]` when needed internally
- derive compatibility `secondaryLearningPoints` from later fix points when needed internally
- parse `nextStepPractice`, `rewriteSuggestions`, `modelAnswer`, and raw candidate pools

`GeminiFeedbackClient.validateGeneratedSections(...)`

Current validation emphasis:

- keep `fixPoints` specific and non-generic
- keep `nextStepPractice` aligned with `fixPoints[0]`
- keep `rewriteSuggestions` aligned with the same optional next step
- keep `modelAnswer` aligned with the same corrected direction
- keep raw candidate pools deduped and useful

## 7. Final UI Assembly

`FeedbackUiComposer.compose(...)`

The final feedback screen is assembled into:

- `ui.fixPoints`
- `ui.nextStepPractice`
- `ui.rewriteSuggestions`
- `ui.secondaryLearningPoints` (mainly expression-style support cards)
- `ui.screenPolicy`
- `ui.loopStatus`

On the frontend, the main correction flow reads from `ui.fixPoints`, not `ui.primaryFix`.

## 8. Frontend Rendering

`apps/frontend/app/answer-loop.tsx`

The modern feedback screen is centered on:

1. keep strengths
2. fix points
3. next-step practice
4. rewrite suggestions
5. model answer
6. expression / reusable language support

The frontend type `FeedbackUi` intentionally does not include `focusCard` or `primaryFix` anymore.

## 9. Practical Rule of Thumb

If you are changing the feedback UX today:

- treat `fixPoints[]` as the source of truth for correction ordering
- treat `nextStepPractice` as the learner-facing optional add-on card
- treat `rewriteSuggestions` as short helper ideas for that same next step
- treat `modelAnswer` as a one-step-up reference answer

If you encounter these fields:

- `summary`
- `inlineFeedback`
- `ui.focusCard`
- `ui.primaryFix`

assume they are legacy / compatibility-oriented unless your target surface explicitly still depends on them.
